GPU Algorithm Simulation Engine

A TypeScript simulation engine that models the GPU algorithm stack from model-level optimization down to warp scheduling and memory — with a live dashboard, pipeline sidebar, and tradeoff scatter plots.

Overview

The engine replays a unified MetricSnapshot stream tick-by-tick through a modular pipeline: model → compute → memory → warp scheduler. Each stage reads the snapshot updated by prior stages so metrics reflect a logical chain.

Data can be synthetic or sourced from MLPerf (when a results JSON is present); other sources fall back to synthetic traces tagged with the selected source.

  • Live dashboard: MFU, HBM util, SM occupancy, loss, bottleneck alerts
  • Pipeline trace per tick (execution order + metric links)
  • Nivo tradeoff graphs: roofline proxy, throughput vs latency, memory vs compute, occupancy vs stalls, speed vs quality

Architecture

Ingestion normalizes external traces into MetricSnapshot[]. The orchestrator runs algorithm modules per tick; SQLite stores snapshots; Express exposes REST + WebSocket; the React dashboard consumes live updates.

Quick start

Clone the repository, install dependencies, and run the dev servers (backend + frontend):

# Install
npm install

# Development (API + Vite dashboard)
npm run dev

Open the app URL shown in the terminal (typically http://localhost:5173). Use Start simulation to begin a run and stream metrics.

Production build

npm run build

Key concepts

Concept Description
MetricSnapshot Normalized metric record per simulation tick
SimTick Clock event passed to each algorithm module
MFU Model FLOP utilization — primary efficiency signal
HBM bandwidth util Memory subsystem pressure
SM occupancy Scheduler saturation on streaming multiprocessors

Deploy this site on GitHub Pages

This folder is a static single-page site (HTML + CSS). To publish it directly from the repository:

  1. Push the docs/ folder to your GitHub branch (e.g. main).
  2. In the repo on GitHub: Settings → Pages → Build and deployment → Source: Deploy from a branch.
  3. Branch: main (or your default), folder: /docs, then Save.
  4. After the workflow finishes, the site is available at https://<username>.github.io/<repo>/

The empty .nojekyll file disables Jekyll processing so paths and assets behave as plain static files.
